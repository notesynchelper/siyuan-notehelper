# 思源笔记同步助手 — 多设备同步防重复方案

## 一、现状分析

### 1.1 当前去重机制

插件已实现**双层去重**策略（`fileHandler.ts`）：

**独立文件模式（MergeMode.NONE）：**

```
Layer 1: 路径去重（getDocumentByHPath）
  └─ SiYuan API: /api/filetree/getIDsByHPath
  └─ 直接查文件系统，不依赖索引
  └─ 命中 → 跳过

Layer 2: 属性去重（checkDocumentBySourceId）
  └─ SQL: SELECT block_id FROM attributes WHERE name='custom-source-id' AND value='{id}'
  └─ 依赖思源数据库索引
  └─ 命中 → 跳过
```

**合并模式（MergeMode.MESSAGES / ALL）：**

```
Step 1: 查找合并目标文档（getDocumentByPath）
  └─ 通过路径 + SQL 查 custom-merge-date 属性
  └─ 找到 → 进入 Step 2
  └─ 找不到 → 创建新合并文档

Step 2: 检查文章是否已合并（getMergedIds）
  └─ 读取块属性 custom-merged-ids（JSON 数组）
  └─ 包含当前 article.id → 跳过
  └─ 不包含 → 追加内容并更新 merged-ids
```

### 1.2 同步时间管理

```typescript
// syncManager.ts:82-95
// 读取 syncAt 并应用时间回溯
effectiveSyncAt = syncDate - syncTimeOffset(默认12小时)

// syncManager.ts:140-143
// 同步完成后立即写入新的 syncAt
this.settings.syncAt = now.toISOString()
await this.plugin.saveSettings()  // 立即写盘，无防抖
```

### 1.3 当前架构的限制

| 机制 | 有效场景 | 失效场景 |
|------|---------|---------|
| 路径去重 | 单设备重复同步、重启后同步 | 另一设备文件尚未同步到本地 |
| 属性去重 | 同设备索引就绪后的重复检查 | 索引延迟、跨设备文件不可见 |
| merged-ids | 同设备同一合并文档内去重 | 另一设备独立创建了合并文档 |
| syncAt | 增量拉取避免重复 | 两设备共享 syncAt 导致互相覆盖 |

## 二、重复场景分析

### 场景 1：两台设备同时同步（高风险）

```
T0: PC 和手机的思源通过云端同步
    两端 syncAt = "2026-02-27T10:00:00Z"

T1 (PC):
    用户打开思源 → 插件触发同步
    → 从 API 拉取文章 A, B, C
    → 本地无对应文档 → 创建 A.md, B.md, C.md
    → 更新 syncAt = "2026-02-28T08:00:00Z"
    → 文件写入本地，等待思源同步上传

T2 (手机，几乎同时):
    用户打开思源 → 插件触发同步
    → syncAt 仍然是 "2026-02-27T10:00:00Z"（PC 的更新还没同步来）
    → 从 API 拉取同样的文章 A, B, C
    → 本地也没有对应文档（PC 的文件还没同步来）
    → 再次创建 A.md, B.md, C.md
    → 更新 syncAt

T3 (同步传播):
    思源同步发现两端都创建了同路径文件
    → 产生冲突，可能出现重复文件或冲突副本
```

**根因**：所有去重检查都基于本地文件系统/数据库查询，对方设备的文件尚未传播时完全失效。

### 场景 2：syncAt 互相覆盖（高风险）

```
T1: PC 同步完成 → syncAt = "2026-02-28T08:00:00Z" → 写入 data.json
T2: 手机同步完成 → syncAt = "2026-02-28T08:05:00Z" → 写入 data.json
T3: 思源同步 data.json → 后写入的覆盖先写入的
T4: PC 读到手机的 syncAt → 丢失了 PC 端 08:00-08:05 之间的增量
    或反之：手机读到 PC 的旧 syncAt → 重新拉取已同步的文章
```

**根因**：全局单一 syncAt 被多设备共享，无法独立追踪各设备的同步进度。

### 场景 3：合并模式下重复创建容器文档（高风险）

```
T1: PC 同步微信消息 → 创建 "同步助手_2026-02-28.md"
    → 追加消息 A, B → custom-merged-ids = ["A", "B"]
T2: 手机同步 → 本地没有该文档（未同步来）
    → 创建另一个 "同步助手_2026-02-28.md"
    → 追加同样的消息 A, B
T3: 同步传播 → 两个同名文档出现冲突
```

### 场景 4：思源同步延迟导致属性查询失败（中等风险）

```
T1: PC 创建文档并设置 custom-source-id
T2: 手机触发同步（PC 的文件刚同步来但索引未完成）
    → getDocumentByHPath：路径存在 ✓（文件已到）
    → 正常跳过

    但如果路径模板不同（用户在两端配置不一致）：
    → getDocumentByHPath：找不到（路径不同）
    → checkDocumentBySourceId：SQL 查询属性表 → 可能找不到（索引延迟）
    → 创建重复文档
```

### 场景 5：merged-ids JSON 解析失败（中等风险）

```typescript
// fileHandler.ts getMergedIds()
const idsJson = await this.getBlockAttribute(docId, 'custom-merged-ids');
const ids = JSON.parse(idsJson);  // 如果 JSON 损坏 → 异常
// catch 块返回空数组 → 所有文章被视为"未合并" → 重复追加
```

**触发条件**：思源同步过程中文件部分写入、属性被截断或编码错误。

### 场景 6：高亮更新不同步（数据丢失风险）

```
T1: PC 同步文章，含高亮 A, B, C
T2: 用户在后端新增高亮 D
T3: PC 再次同步 → 文章 ID 已存在 → 跳过
    → 高亮 D 永远不会被同步到本地
```

**根因**：去重是文章级别的，不检查文章内容是否有更新。

## 三、防重复优化方案

参考 `gate/obsidian-plug/防重复优化方案.md`，针对思源笔记的 API 和存储机制进行适配设计。

### 改动一：全局 ID 索引扫描

#### 改动内容

在同步循环开始前，通过思源 SQL API 一次性查询所有已同步文章的 source-id 和 merged-ids，构建内存索引 Map。处理每篇文章前先查此索引，已存在则跳过或更新。

#### 详细设计

**新增：`src/sync/idIndex.ts`**

```typescript
import { logger } from '../utils/logger';

/**
 * 全局 ID 索引
 * 在同步开始前一次性构建，用于跨设备去重
 */
export class IdIndex {
    // articleId → blockId 映射
    private index: Map<string, string> = new Map();

    /**
     * 从思源数据库构建索引
     * 查询所有 custom-source-id 和 custom-merged-ids 属性
     */
    async build(): Promise<void> {
        const startTime = Date.now();

        // 1. 查询独立文件的 source-id
        await this.loadSourceIds();

        // 2. 查询合并文档的 merged-ids
        await this.loadMergedIds();

        logger.info(
            `[IdIndex] 索引构建完成: ${this.index.size} 个 ID, ` +
            `耗时 ${Date.now() - startTime}ms`
        );
    }

    /**
     * 查询所有 custom-source-id 属性
     */
    private async loadSourceIds(): Promise<void> {
        const sql = `SELECT block_id, value FROM attributes WHERE name='custom-source-id'`;
        try {
            const response = await fetch('/api/query/sql', {
                method: 'POST',
                headers: { 'Content-Type': 'application/json' },
                body: JSON.stringify({ stmt: sql }),
            });
            const result = await response.json();
            if (result.code === 0 && Array.isArray(result.data)) {
                for (const row of result.data) {
                    if (row.value && row.block_id) {
                        this.index.set(row.value, row.block_id);
                    }
                }
                logger.debug(`[IdIndex] 加载了 ${result.data.length} 个 source-id`);
            }
        } catch (error) {
            logger.warn('[IdIndex] 加载 source-id 失败:', error);
        }
    }

    /**
     * 查询所有 custom-merged-ids 属性并展开
     */
    private async loadMergedIds(): Promise<void> {
        const sql = `SELECT block_id, value FROM attributes WHERE name='custom-merged-ids'`;
        try {
            const response = await fetch('/api/query/sql', {
                method: 'POST',
                headers: { 'Content-Type': 'application/json' },
                body: JSON.stringify({ stmt: sql }),
            });
            const result = await response.json();
            if (result.code === 0 && Array.isArray(result.data)) {
                let count = 0;
                for (const row of result.data) {
                    if (!row.value || !row.block_id) continue;
                    try {
                        const ids = JSON.parse(row.value);
                        if (Array.isArray(ids)) {
                            for (const id of ids) {
                                this.index.set(String(id), row.block_id);
                                count++;
                            }
                        }
                    } catch {
                        logger.warn(`[IdIndex] 解析 merged-ids 失败: ${row.block_id}`);
                    }
                }
                logger.debug(`[IdIndex] 加载了 ${count} 个 merged-id`);
            }
        } catch (error) {
            logger.warn('[IdIndex] 加载 merged-ids 失败:', error);
        }
    }

    /**
     * 通过文章 ID 查找是否已存在
     */
    findBlockId(articleId: string): string | undefined {
        return this.index.get(articleId);
    }

    /**
     * 获取索引大小
     */
    get size(): number {
        return this.index.size;
    }
}
```

#### 性能影响评估

| 维度 | 影响 | 详细说明 |
|------|------|----------|
| **索引构建时间** | +10-50ms | 两条 SQL 查询，思源 SQLite 本地查询很快。100 条记录 <10ms，1000 条 <30ms，10000 条 <50ms |
| **内存占用** | 极小 | Map 存储字符串键值对。1000 条 ≈ 80KB |
| **每条 item 查找** | O(1) | Map.get() 哈希查找，<0.01ms |
| **与现有去重的关系** | 正向 | ID 索引命中时直接跳过，省去后续的 API 调用（getDocumentByHPath + checkDocumentBySourceId），每条省 5-20ms |

**结论：净正向效果，尤其在重复同步场景下显著减少 API 调用。**

#### 与 Obsidian 方案的差异

| Obsidian 方案 | 思源方案 | 差异原因 |
|------|------|----------|
| 使用 `metadataCache.getFileCache()` 遍历所有 md 文件 | 使用 SQL 查询 attributes 表 | 思源的 SQLite 数据库比遍历文件系统更高效 |
| 解析 front matter 的 `id` 字段 | 查询 `custom-source-id` 块属性 | 思源用块属性而非 front matter 存储元数据 |
| 解析 front matter 的 `messages` 数组 | 查询 `custom-merged-ids` 块属性 | 同上 |
| 可能受 metadataCache 延迟影响 | SQL 查询直接读数据库，不受缓存延迟影响 | 思源的 SQL API 直接操作数据库 |

### 改动二：设备级独立游标

#### 改动内容

为每台设备生成唯一 ID，在插件 settings 中以 `deviceSyncCursors: { [deviceId]: syncAt }` 的形式存储各设备独立的同步游标。同步时只读写自己的游标，不干扰其他设备。

#### 详细设计

**1. `src/settings/index.ts` — 新增字段**

```typescript
export interface PluginSettings {
    // ... 现有字段

    // 新增：设备级同步游标
    deviceSyncCursors: Record<string, string>;
}

export const DEFAULT_SETTINGS: PluginSettings = {
    // ... 现有字段

    // 新增
    deviceSyncCursors: {},
};
```

**2. `src/sync/syncManager.ts` — 设备 ID 生成**

```typescript
/**
 * 获取当前设备的唯一标识
 * 使用 localStorage 持久化（不跨设备同步，每台设备独有）
 */
private getDeviceId(): string {
    const STORAGE_KEY = 'notehelper-device-id';
    try {
        let id = window.localStorage.getItem(STORAGE_KEY);
        if (!id) {
            // 生成设备ID: 平台前缀 + 时间戳 + 随机字符串
            const platform = (window as any).siyuan?.config?.system?.os === 'android'
                || (window as any).siyuan?.config?.system?.os === 'ios'
                ? 'mobile' : 'desktop';
            id = `${platform}-${Date.now().toString(36)}-${Math.random().toString(36).substring(2, 6)}`;
            window.localStorage.setItem(STORAGE_KEY, id);
            logger.info(`[SyncManager] 生成新设备ID: ${id}`);
        }
        return id;
    } catch {
        // localStorage 不可用时使用临时 ID
        return `temp-${Math.random().toString(36).substring(2, 8)}`;
    }
}
```

> **为什么用 localStorage？**
> - localStorage 是浏览器/Electron 级别的本地存储，**不会**被思源同步到其他设备
> - 每台设备有独立的 localStorage 实例
> - 即使用户清除数据，新 ID 的游标为空 → 回退到全局 syncAt → 不会丢数据

**3. `src/sync/syncManager.ts` — sync() 中 syncAt 读写改造**

```typescript
async sync(): Promise<SyncResult> {
    // ...

    const deviceId = this.getDeviceId();

    // 读取 syncAt 时优先使用自己的设备游标
    const rawSyncAt = this.settings.deviceSyncCursors?.[deviceId]
        || this.settings.syncAt  // 回退到全局（兼容升级）
        || '';

    // 计算有效同步时间（应用时间回溯）
    if (rawSyncAt) {
        const syncDate = new Date(rawSyncAt);
        const offsetHours = this.settings.syncTimeOffset || 0;
        if (offsetHours > 0) {
            syncDate.setHours(syncDate.getHours() - offsetHours);
            effectiveSyncAt = syncDate.toISOString().replace(/\.\d{3}Z$/, 'Z');
        } else {
            effectiveSyncAt = rawSyncAt;
        }
    }

    // ... 处理文章 ...

    // 同步完成后更新游标
    const now = new Date();
    const nowStr = now.toISOString().replace(/\.\d{3}Z$/, 'Z');

    // 更新全局游标（向后兼容）
    this.settings.syncAt = nowStr;

    // 更新设备级游标
    if (!this.settings.deviceSyncCursors) {
        this.settings.deviceSyncCursors = {};
    }
    this.settings.deviceSyncCursors[deviceId] = nowStr;

    // 清理过期游标（超过 30 天未更新的设备）
    this.cleanStaleDeviceCursors();

    await this.plugin.saveSettings();  // 立即写盘
}
```

**4. 过期游标清理**

```typescript
private cleanStaleDeviceCursors(): void {
    const cursors = this.settings.deviceSyncCursors;
    if (!cursors) return;
    const thirtyDaysAgo = new Date();
    thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30);

    for (const [deviceId, cursor] of Object.entries(cursors)) {
        try {
            const cursorTime = new Date(cursor);
            if (cursorTime < thirtyDaysAgo) {
                delete cursors[deviceId];
                logger.debug(`[SyncManager] 清理过期设备游标: ${deviceId}`);
            }
        } catch {
            // 无效日期格式，也清理
            delete cursors[deviceId];
        }
    }
}
```

**5. `resetSyncTime()` 适配**

```typescript
async resetSyncTime(): Promise<void> {
    this.settings.syncAt = '';

    // 同时重置当前设备的游标
    const deviceId = this.getDeviceId();
    if (this.settings.deviceSyncCursors) {
        this.settings.deviceSyncCursors[deviceId] = '';
    }

    await this.plugin.saveSettings();
    logger.debug('Sync time reset (including device cursor)');
}
```

| **API 调用** | 无变化 | 只改变传给 API 的 syncAt 参数值 |

#### 兼容性评估

| 维度 | 影响 |

| **思源同步的 data.json 合并** | 思源同步对插件 data.json 是整文件同步，可能出现覆盖。但因为保留了全局 `syncAt`，最坏情况仅回退到当前行为 |

### 改动三：合并文档的 source-id 属性检查增强

#### 改动内容

当前合并模式的去重只依赖 `custom-merged-ids` JSON 数组。增加一个前置检查：在合并前也通过 `custom-source-id` SQL 查询确认文章是否已在 vault 中存在（可能在另一个合并文档中）。

#### 详细设计

此改动已被**改动一（全局 ID 索引）**覆盖。全局索引同时加载了 `custom-source-id` 和 `custom-merged-ids`，在 `mergeArticleToFile()` 入口即可拦截。

无需额外代码。

## 四、三个方案的协同效果

```
设备 A 同步：
1. IdIndex.build() — SQL 查询所有已有 ID（改动一）
2. 读取 deviceSyncCursors["desktop-a3f2c1"]（改动二）
3. 从 API 增量拉取数据
4. 对每篇文章：
   a. idIndex.findBlockId(article.id) → 未命中
   b. 路径去重 → 未命中
   c. 属性去重 → 未命中
   d. 创建文档 + 设置 custom-source-id
5. 更新 deviceSyncCursors["desktop-a3f2c1"]（改动二）

设备 B 同步（在设备 A 之后几分钟，文件已同步来）：
1. IdIndex.build() — 扫描到 A 创建的文档的 source-id（改动一）
2. 读取 deviceSyncCursors["mobile-b7e4d9"]（独立于 A）（改动二）
3. 从 API 拉取数据（可能和 A 相同）
4. 对每篇文章：
   a. idIndex.findBlockId(article.id) → 命中！→ 跳过
5. 无重复创建
```

**最坏情况（A 和 B 几乎同时同步，文件尚未传播）：**
- 两台设备各自创建了相同 ID 的文档
- 思源同步传播后，vault 中出现重复
- 下次同步时，ID 索引发现重复 → 不再继续创建 → 问题不会持续恶化
- 用户只需手动删除一份重复文件（一次性问题）

## 五、综合性能影响

| 改动 | 同步启动 | 每条 item | 同步结束 | 内存 |
|------|---------|----------|---------|------|
| ID 索引构建 | +10-50ms（一次性） | -5-20ms（命中时跳过 API 调用） | 无 | +数十 KB |
| 设备游标 | +<1ms | 无 | +<1ms | +<1KB |
| **合计** | +10-50ms | 净正向 | +<1ms | +数十 KB |

**用户感知**：
- 小型数据（<100 篇文章）：完全无感
- 中型数据（100-1000 篇）：同步启动多 10-30ms，后续处理更快
- 大型数据（>1000 篇）：同步启动多 30-50ms，SQL 查询仍然很快

## 六、对现有功能的负向影响

| 风险点 | 概率 | 严重度 | 缓解措施 |
|--------|------|--------|----------|
| SQL 查询在思源刚启动时数据库未就绪 | 低 | 低 | 已有 syncOnStart 延迟；IdIndex 构建失败不阻塞同步，仅失去索引去重层 |
| localStorage 不可用 | 极低 | 低 | try-catch 回退到临时 ID |
| 思源同步覆盖 data.json 导致设备游标丢失 | 中 | 低 | 保留全局 `syncAt` 作为 fallback；丢失游标最多导致多拉一次数据（被 ID 索引拦截） |
| resetSyncTime 未适配 | 高（如果漏改） | 中 | 方案中已包含适配代码 |

**结论：所有风险都有缓解措施，最坏情况只是回退到当前行为（不会比现在更差）。**

## 七、实施优先级

| 顺序 | 改动 | ROI | 理由 |
|------|------|-----|------|
| 1 | 全局 ID 索引 | 最高 | 直接解决跨设备去重的核心问题，两条 SQL 查询即可覆盖所有已同步文章 |
| 2 | 设备级独立游标 | 高 | 从根源上避免 syncAt 互相覆盖导致的重复拉取 |

## 八、验证方案

### 单元测试

- `IdIndex.build()` — mock SQL 响应，验证 source-id 和 merged-ids 的加载
- `IdIndex.findBlockId()` — 验证查找逻辑
- `getDeviceId()` — 验证首次生成和二次读取
- `deviceSyncCursors` 的读取回退逻辑（无设备游标 → 全局 syncAt → 空字符串）

### 集成测试场景

1. **首次升级**：无 deviceSyncCursors → 回退到全局 syncAt → 正常同步
2. **单设备重复同步**：创建文档 → 再次同步 → ID 索引命中 → 跳过
3. **resetSyncTime**：重置后 deviceSyncCursors 和 syncAt 都为空 → 全量拉取
4. **模拟跨设备**：手动创建含 custom-source-id 属性的文档 → 同步时 ID 索引命中 → 不重复创建

### 手动验证

1. PC 端同步 → 检查 data.json 中 deviceSyncCursors 有 desktop-xxx
2. 手机端同步 → 检查 data.json 中 deviceSyncCursors 有 mobile-xxx
3. 两端各自同步 → 确认无重复文档产生
4. 重置同步时间 → 全量拉取 → 确认已有文档被 ID 索引跳过
